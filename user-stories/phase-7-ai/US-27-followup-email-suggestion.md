# US-27 — AI-Drafted Follow-up Email Suggestion

---

**User Story: AI-Drafted Follow-up Email Suggestion**

**As a** document owner reviewing a viewer's engagement,
**I want** an AI-drafted follow-up email suggestion tailored to a specific viewer's engagement,
**So that** I know what to say next instead of guessing, based on how much of the document they actually read.

**Acceptance Criteria:**

- [ ] A new authenticated endpoint `POST /api/v1/documents/[id]/links/[linkId]/followup-email` exists
- [ ] Request body: `{ viewer_fingerprint: string }` — identifies which row from the per-viewer engagement list (US-06, already shipped in Phase 7) to draft for
- [ ] Returns `403` if the requesting user does not own the document; `404` if the document, link, or that specific viewer (no `link_views` row for that fingerprint on this link) doesn't exist
- [ ] Returns `200` with `{ subject: string, body: string }` — a short, professional follow-up email draft in Brazilian Portuguese (pt-BR)
- [ ] The draft is generated by the Claude API from the viewer's own engagement data (already computed by `models/linkView.ts#getViewersByLinkId`): `engagement_score`, `total_time_on_page`, `max_pages_viewed` vs. the document's `page_count`, `visit_count`, `downloaded`, and `viewer_name`/`viewer_email` if known
- [ ] The draft references at least one concrete signal from that data (e.g., "percebi que você chegou até a página 3 de 10" or "vi que você baixou o arquivo mas ainda não recebi seu retorno") — a generic template with no personalization fails review
- [ ] If the viewer has no recorded `viewer_email`, the draft is still generated (the owner may know the person outside the platform) but the response includes `viewer_email: null` so the frontend can show a "no address on file" hint instead of a mailto link
- [ ] The draft is **not** persisted to the database or sent automatically — it's copy the owner reviews and sends themselves from their own mail client. Copy-paste is enough for v1; no "Send" button.
- [ ] Rate limit: max 20 requests per authenticated user per hour across this endpoint (protect API costs; deliberately generous since this is a manual, one-off action per viewer rather than something a UI could loop over automatically)
- [ ] Returns a `503 ServiceError` with a clear message if `ANTHROPIC_API_KEY` is not configured, rather than a generic 500 — this is a synchronous, user-initiated action, unlike the fire-and-forget summarization/insights jobs, so silently no-op'ing would be a worse experience than a clear "AI feature unavailable" message
- [ ] In `LinkAnalyticsDrawer.tsx`, each row in the existing viewer engagement list (`ViewerEngagementList.tsx`) gets a "Sugerir e-mail" button that calls this endpoint and displays the result (subject + body, with a "Copiar" button) inline or in a small popover — no new modal needed
- [ ] Integration tests cover: 403 for non-owner, 404 for nonexistent link/viewer, 200 with a well-formed `{ subject, body }` for a seeded viewer, rate limit enforced after 20 calls/hour, and a `503` when `ANTHROPIC_API_KEY` is unset

**Technical Context:**

- Relevant files:
  - `pages/api/v1/documents/[id]/links/[linkId]/followup-email/index.ts` _(create)_
  - `models/followupEmail.ts` _(create — builds the prompt from a single viewer's engagement row and calls the shared AI client)_
  - `models/linkView.ts` _(add a `getViewerByFingerprint(linkId, fingerprint)` — same aggregation query as `getViewersByLinkId` but filtered to one viewer, reused by the new model)_
  - `types/index.ts` _(add `FollowUpEmailSuggestion` interface; add `viewer_fingerprint` to `ViewerEngagement` so the frontend has something to send back — it's a hashed per-browser fingerprint, not sensitive to the document owner who already sees the rest of this viewer's activity)_
  - `components/analytics/ViewerEngagementList.tsx` _(add the "Sugerir e-mail" button + inline result display)_
- Prompt structure: system = "Você é um assistente que ajuda donos de documentos a escrever e-mails de follow-up curtos e naturais, em português do Brasil, baseados em dados reais de engajamento do leitor. Nunca invente informações que não estão nos dados fornecidos." User message = the viewer's engagement JSON (score, time, pages read vs. total, visits, downloaded, name/email if known) + document title + "Escreva um e-mail de follow-up curto (assunto + corpo) citando pelo menos um dado concreto de engajamento. Responda como JSON válido: `{ subject: string, body: string }`."
- Reuses the shared AI client set up for US-14/15/17/18 (`infra/ai.ts`) and the same `claude-haiku-4-5-20251001` model for cost efficiency — this is a short, low-complexity generation task, same tier as the analytics insight summary.
- Depends on: Phase 7's engagement score (`models/linkView.ts#computeEngagementScore`, already shipped) and, loosely, US-17/US-18's insight/suggestion endpoints (not a hard dependency — this endpoint is independently callable — but shares the same AI client setup, so building it after US-17/18 avoids duplicating that plumbing).
- Dependencies / considerations:
  - Requires `infra/ai.ts` (shared Anthropic client, added alongside US-14)
  - No new migration needed — this reads existing `link_views` data and returns an ephemeral result, nothing new to persist
