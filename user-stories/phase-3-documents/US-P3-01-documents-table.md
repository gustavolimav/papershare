# US-P3-01 — Documents Table & Migration

---

**User Story: Documents Table**

**As a** backend developer setting up document storage,
**I want to** create the `documents` database table and corresponding TypeScript types,
**So that** the system has a persistent, queryable store for document metadata with soft-delete support.

**Acceptance Criteria:**

- [ ] A migration file `005-create-documents.sql` exists in `infra/migrations/`
- [ ] The migration creates a `documents` table with the following columns:
  - `id UUID PRIMARY KEY` — generated by the application (not `gen_random_uuid()`) to allow the storage key to reference the same UUID
  - `title VARCHAR(255) NOT NULL`
  - `description VARCHAR(1000)` — nullable
  - `original_filename VARCHAR(255) NOT NULL` — the user's original file name
  - `storage_key VARCHAR(512) NOT NULL UNIQUE` — the key used to retrieve the file from storage
  - `mime_type VARCHAR(100) NOT NULL`
  - `size_bytes BIGINT NOT NULL`
  - `page_count INTEGER` — nullable; extracted from PDF on upload
  - `user_id UUID NOT NULL REFERENCES users(id)`
  - `created_at TIMESTAMPTZ DEFAULT timezone('utc', now())`
  - `updated_at TIMESTAMPTZ DEFAULT timezone('utc', now())`
  - `deleted_at TIMESTAMPTZ` — nullable; set on soft-delete
- [ ] TypeScript interfaces added to `types/index.ts`:
  - `Document` — full DB row
  - `DocumentCreateInput` — input for creating a document (title, description, file metadata)
  - `DocumentUpdateInput` — input for patching title and/or description
  - `DocumentResponse` — public-safe shape (same as `Document` minus internal fields if any)
- [ ] Migration runs cleanly via `npm run migrations:up` with no errors
- [ ] An index on `user_id` exists for fast per-user document queries

**Technical Context:**

- Relevant files:
  - `infra/migrations/005-create-documents.sql` *(create)*
  - `types/index.ts` *(add Document interfaces)*
- The `id` column intentionally does NOT use `DEFAULT gen_random_uuid()` because the application generates the UUID first and uses it as the basis for the `storage_key` filename (e.g., `<uuid>.pdf`)
- Soft-delete pattern: all queries must filter `WHERE deleted_at IS NULL` — never hard-delete rows
- Dependencies / considerations:
  - Must run after migration 004 (`users` table with `deleted_at` column must exist)
  - `user_id` references `users(id)` — no `ON DELETE CASCADE` intentionally; deleting a user is a soft-delete, so the documents remain
