## [Unreleased]

### Fixed

- `ServiceError`'s constructor accepted an `action` field in its options type but silently ignored it, always hardcoding the same generic "Verifique se o serviço está disponível." — so the specific, actionable message the new BYOK follow-up-email feature needs ("Peça para o dono do documento configurar sua chave de IA em Configurações.") never actually reached the response. Caught via manual verification in-browser. Every other custom error class in `infra/errors.ts` already respected a passed-in `action`; `ServiceError` was the one exception. Existing `ServiceError` call sites (`infra/storage.ts`, `infra/database.ts`) don't pass `action` explicitly, so they keep the same generic fallback message as before.
- `ServiceError` (503) instances thrown anywhere in the app — including the new AI-unavailable case below, but also pre-existing ones in `infra/storage.ts` and `infra/database.ts` — fell through `infra/controller.ts#onErrorHandler`'s `instanceof` chain (which checked every other custom error class except this one) into the generic `InternalServerError` fallback. The response's status code happened to still come out as 503 (the fallback forwards `err.statusCode`), but the specific, actionable `message`/`action` were replaced with the generic "Um erro interno não esperado aconteceu." — masked in the browser as an unhelpful error. Caught via manual verification of the new follow-up-email feature's "AI unavailable" message. Added the missing `err instanceof ServiceError` branch.
- Custom branding's accent color (`brand_accent_color`) correctly overrode the `--primary` CSS variable on the ready-state wrapper, but no element in the actual PDF viewing experience used `bg-primary`/`text-primary` — every toolbar button in `components/viewer/ViewerControls.tsx` used the `ghost`/`outline` variants, so the color was wired up but never visible for the common case (viewing a PDF). Changed the toolbar's "Download" button to the `default` variant (`bg-primary`), so it now visibly picks up the link's accent color when set, and falls back to the normal theme color otherwise (verified no visual change for links without branding configured).
- Vercel build warned `Node.js version 20.x is deprecated. Deployments created on or after 2026-10-01 will fail to build.` Bumped `package.json#engines`, `.nvmrc`, and both GitHub Actions workflows from Node 20 (`lts/iron`) to `24.x`, matching Vercel's own suggested fix, so local dev, CI, and Vercel all build on the same, still-supported major.
- `/documents/[id]/analytics` (the per-link/per-page analytics dashboard, including the page-view heatmap) was fully built and reachable by direct URL, but nothing in the UI linked to it — the document detail page had no way to navigate there. Added an "Analytics" button next to "Editar" in `components/documents/DocumentMeta.tsx`.
- Vercel build warned that `package.json`'s `"engines": { "node": ">=20.16.0" }` is open-ended, so it overrides the Project Settings dropdown ("22.x") and silently jumps to whatever the newest major Node release is (24.x at the time) instead — meaning a future Node major release could change the runtime under this app with no code change and no warning beyond a build log line. Bounded the range to `>=20.16.0 <21.0.0`, matching `.nvmrc` (`lts/iron`) and the GitHub Actions workflows, so the version actually used is explicit and stable across local dev, CI, and Vercel.

- CI (`Jest Ubuntu`) was failing on every document-upload-dependent test with "Unexpected token < in JSON": `@aws-sdk/client-s3` requires Node `>=20.0.0` and `pdf-parse`/`pdfjs-dist` require Node `>=20.16.0`, but CI and `.nvmrc` were pinned to `lts/hydrogen` (Node 18) — confirmed via `npm warn EBADENGINE` in the CI log. Bumped `.nvmrc` and both GitHub Actions workflows to `lts/iron` (Node 20), and added an `engines` field to `package.json` so this fails fast with a clear message instead of an obscure runtime crash next time.
- Public PDF viewer (`components/viewer/PDFViewer.tsx`) threw `TypeError: Object.defineProperty called on non-object` on every load, from inside `pdfjs-dist`'s own bundled module code. Confirmed via a native `<script type="module">` test that the unmodified `pdf.mjs` works fine in the same browser — the crash only happens when Next's webpack re-bundles the package's already-webpack-bundled `.mjs`, corrupting its internal export wiring. Fixed by loading both the library and its worker as real native ES modules straight from the jsdelivr CDN (`import(/* webpackIgnore: true */ ...)`), instead of importing the npm package.
- Vercel's production build (`next build`) failed with 26 real ESLint errors across the new frontend files (`'React' is not defined` / `no-undef`, `'x' is defined but never used` / `no-unused-vars`) that never showed up locally in `npm run sf`. Root cause: `.eslintrc.json` was missing `"root": true`, so ESLint's legacy config resolution walked up past the project into an ancestor directory, found another copy of the same plugin, and hard-aborted linting entirely — silently no-op'ing every local lint run without failing the command. Separately, `eslint:recommended`'s base `no-undef`/`no-unused-vars` (extended after `next/core-web-vitals`) were clobbering Next's TypeScript-aware versions of those rules, which don't understand type-only positions (e.g. a named parameter in a function _type_ signature, or a bare `React.FormEvent` type reference) and flag them as real undefined/unused variables. Fixed by adding `"root": true` and an override enabling `@typescript-eslint/no-unused-vars` (and disabling the base rules) for `*.ts`/`*.tsx` files — the standard fix recommended by typescript-eslint for this exact class of false positive.
- `GET /api/v1/documents` (dashboard document listing) 500'd on every request on Vercel with `ReferenceError: DOMMatrix is not defined`, thrown from inside `pdfjs-dist`'s legacy Node build. Cause: `pdf-parse` was imported at the top of `pages/api/v1/documents/index.ts`, a single file that backs both the `GET` (list) and `POST` (upload) handlers as one serverless function — so evaluating that top-level import ran on every request regardless of method, and `pdfjs-dist`'s legacy build references browser canvas globals (`DOMMatrix`) that only exist if a native canvas binary (`@napi-rs/canvas`) loads successfully, which apparently didn't happen on Vercel's Linux runtime even though it worked locally on macOS. Fixed by lazily `import()`-ing `pdf-parse` only inside `extractPageCount()`, so listing/deleting/etc. documents never touches it.
  - That alone wasn't enough: actual PDF uploads still 500'd, because `pdfjs-dist`'s internal geometry code throws the same `DOMMatrix` error in a way that isn't reliably caught by a try/catch around the awaited call (not tied to the promise `getInfo()` returns — looks like an internal fire-and-forget operation). Fixed for real by probing `@napi-rs/canvas` with its own `import()` first — a plain, catchable module-resolution failure — and skipping page-count extraction entirely (falling back to `page_count: null`) when that probe fails, instead of ever calling into `pdf-parse` without it.
- Downloading a password-protected share link always returned `403 Senha incorreta`, even right after typing the correct password to view it. Cause: the download button (`components/viewer/PDFViewer.tsx`) and the non-PDF fallback link (`components/viewer/ViewerPage.tsx`) both pointed the browser straight at `/api/v1/share/[token]/file` via `window.open`/`<a href>` — a plain navigation can't attach the `X-Share-Password` header the route requires, so it always looked like no password was supplied. Fixed by downloading through `fetch()` with the header instead: PDFs reuse the bytes already fetched for the viewer (no extra request), other file types fetch fresh with the remembered password, and either way the bytes are handed to the browser as a `Blob` object URL via a programmatically-clicked anchor.

### Added

- Phase 9 (Team Workspaces, "workspaces básico" slice) — US-30, member invitation & role management: `GET`/`POST /api/v1/workspaces/[id]/members` (list every member with their role, invite by email — existing accounts only, no invite-token/accept flow) and `PATCH`/`DELETE /api/v1/workspaces/[id]/members/[userId]` (change role, remove a member, or leave — `DELETE` on one's own `userId` doubles as "leave workspace" and is allowed at any role, unlike removing someone else which requires `owner`). Invite returns a specific `404` ("Nenhuma conta encontrada com esse email...") when no account matches, distinct from a generic not-found, since this is lead-invitation not a lookup; `409` if the email is already a member. New `ConflictError` (409) added to `infra/errors.ts`/`infra/controller.ts` — the codebase's error classes didn't have one yet. Both role-change and removal share a last-owner guard (`models/workspace.ts#assertNotLastOwner`): a workspace's sole `owner` can't be demoted or removed/leave, since that would strand the workspace with no one able to manage it — checked only when the change would actually result in zero owners, so a no-op "owner → owner" reassignment is never blocked even for the only owner. Every membership mutation 403s on the personal workspace, same as US-29's workspace-level mutations.
- Phase 9 (Team Workspaces, "workspaces básico" slice) — US-29, workspace CRUD + switching: `POST`/`GET /api/v1/workspaces` (create a team workspace — the requester becomes its `owner`; list every workspace the requester belongs to, personal one first, each with the requester's own `role`), `PATCH`/`DELETE /api/v1/workspaces/[id]` (rename/soft-delete, `owner`-only, `403` on the personal workspace since it can't be renamed or deleted), and `POST /api/v1/workspaces/[id]/activate` (switches `users.active_workspace_id`, `404` if the requester isn't a member — same response whether the workspace doesn't exist, is deleted, or belongs to someone else, so as not to leak which workspaces exist). New `models/workspace.ts#requireRole()` is the central authorization check every later workspace-scoped story (US-30/US-31) builds on: role hierarchy `owner` > `editor` > `viewer`, `NotFoundError` for a missing/deleted/non-member workspace, `ForbiddenError` for an insufficient role. `GET /api/v1/sessions` now also returns `active_workspace_id`, so the frontend knows the active workspace on every page load without an extra request. No frontend yet — API only, the switcher UI lands in US-33.
- Phase 9 (Team Workspaces, "workspaces básico" slice) — US-28, workspace data model: every user now belongs to an automatically-created personal workspace (`is_personal = true`) rather than owning documents directly, laying the foundation for shared team workspaces in the stories that follow (US-29 through US-33). New `workspaces` (migration `024`) and `workspace_members` (migration `025`, role `owner`/`editor`/`viewer`) tables; `documents.workspace_id` (migration `026`) backfills every existing document into its uploader's new personal workspace, and `users.active_workspace_id` (migration `027`) backfills to the same. `models/user.ts#create()` now creates the personal workspace + owner membership + active workspace pointer atomically with the user row itself, on a dedicated transaction client — a user with no personal workspace would be a broken state. `documents.workspace_id` is deliberately left nullable for now (not yet populated on new uploads, and not yet used for any authorization decision) — US-31 applies `NOT NULL` once `models/document.ts#create()` actually populates it; getting this sequencing wrong during development briefly 503'd every document upload (`null value in column "workspace_id" violates not-null constraint`), caught via manual verification before landing.
- Phase 8 (AI Features) — closes out the phase with all six planned stories, sharing one Anthropic client wrapper (`infra/ai.ts`, `claude-haiku-4-5-20251001`) built around a bring-your-own-key model rather than a single platform-wide credential: every AI feature runs against the specific document owner's own Anthropic key (`models/user.ts#getAiApiKey()`), which each user pastes into a new "IA" section on `/settings` (`components/settings/AiSettingsForm.tsx`) and never sees again afterward — the API only ever returns `{ configured: boolean }`, never the key itself. Stored encrypted at rest (`infra/encryption.ts`, AES-256-GCM via a new `ENCRYPTION_KEY` env var — a real reversible credential, unlike `PEPPER`'s one-way password hashing) in a new `users.ai_api_key_encrypted` column (migration `023`). Features degrade the same way as before, just resolved per-owner instead of from a global env var: fire-and-forget ones (summarization, insights) silently no-op when the owner hasn't configured a key; synchronous ones (chat, follow-up email) throw a `ServiceError` (503) with a specific "peça para o dono do documento configurar sua chave" message. A shared `ai_usage_log` table (migration `020`) backs a generic, DB-based rate limiter (`models/aiUsage.ts`) for every cost-incurring action — unlike the existing IP-based `infra/rate-limit.ts` (production-only, used for the public chat endpoint), this one works in every environment and is scoped per-user or per-document depending on the feature.
  - **Auto-summarization on upload**: migration `019-add-ai-summary-to-documents.sql` adds `documents.ai_summary`/`ai_summary_generated_at`. `models/summarizer.ts` extracts PDF text (`models/textExtraction.ts`, PDF-only for this pass — DOCX/PPTX extraction is a documented future addition) and asks Claude for a 2-5 sentence summary; triggered fire-and-forget right after upload (same pattern as the existing new-viewer notification email — not awaited, errors swallowed, response unaffected), reusing the already-in-memory upload buffer instead of re-fetching from storage.
  - **Summary endpoint**: `GET`/`POST /api/v1/documents/[id]/summary` — GET returns the current summary; POST regenerates synchronously (a single Haiku call finishes well within a normal request), rate-limited to 3/hour per user.
  - **Analytics insights**: `GET /api/v1/documents/[id]/analytics/insights` (`models/analyticsInsights.ts`) asks Claude for a 3-5 sentence natural-language summary of a document's analytics, cached on the `documents` row (migration `021`) and only regenerated when `total_views`/`last_viewed_at` change since the last generation — rate-limited to 10 regenerations/day per document (cache hits don't count). Returns a friendly "Nenhuma visualização ainda..." message with no views yet, and quietly falls back to the existing cache if the AI client isn't available.
  - **Drop-off suggestions**: the same insights endpoint asks Claude for up to 3 actionable suggestions (`drop_off` or `engagement` type) when there are ≥5 views, based on real per-page data rather than the `avg_pages_viewed` estimate the original spec assumed — `models/linkView.ts#getPageBreakdownByDocumentId()` aggregates `link_view_pages` across every link of the document (they all point at the same file, so combining them is meaningful, unlike combining across different documents), an improvement made possible by Phase 7's per-page tracking landing after this story was originally scoped.
  - **AI-drafted follow-up email suggestion**: new story, spec written this session (`user-stories/phase-7-ai/US-27-followup-email-suggestion.md`) since it depends on Phase 7's engagement score. `POST /api/v1/documents/[id]/links/[linkId]/followup-email` (`models/followupEmail.ts`) drafts a subject+body referencing at least one concrete engagement signal (page reached, download, visit count) for a specific viewer, identified by the `viewer_fingerprint` now exposed on `ViewerEngagement` (Phase 7's per-viewer list). Ownership/link/viewer-existence checks and the 20/hour rate limit all run _before_ the AI-availability check, so a request from someone without access gets the correct 403/404, not a misleading 503. Never persisted or auto-sent — the owner copies and sends it themselves. New "Sugerir e-mail" button on each row of `ViewerEngagementList.tsx`.
  - **Viewer chat (RAG)**: `POST /api/v1/share/[token]/chat` — public, gated exactly like the sibling share routes (password/email/NDA via `shareLink.getByToken`) before any AI work happens, then streams a Claude response via Server-Sent Events. `models/documentChunks.ts` lazily extracts and stores one chunk per PDF page (migration `022-create-document-chunks.sql`) on first use — most links are never chatted with, so this avoids doing it unconditionally at upload time — and retrieves the most relevant pages for a question via plain keyword-overlap scoring, no vector DB/embeddings pipeline for this MVP. New floating "Perguntar sobre este documento" toggle and `components/viewer/ChatPanel.tsx` on the PDF viewer, reading the SSE stream via `response.body.getReader()` (not `EventSource`, since this needs a POST body and custom auth headers).
  - **Hide AI features when unconfigured**, following up on the BYOK switch above: rather than showing an AI feature and having it fail (fire-and-forget silent no-op, or a 503 on a synchronous action), the four AI-feature entry points now don't render at all unless the relevant key is present. `lib/useAiKeyConfigured.ts` (new) is a small SWR hook that reads the logged-in user's own status from the same `/api/v1/users/[username]/ai-key` cache key `AiSettingsForm.tsx` already uses, so saving or removing a key in Settings instantly reflects everywhere else it's read — no separate global-state layer needed. Applied to `SummaryCard.tsx` (hides the whole card, even if a summary was generated before the key was removed) and `InsightCard.tsx` (also makes its SWR fetch itself conditional, avoiding a wasted network call) on the document/analytics owner-facing pages, and to the "Sugerir e-mail" button in `ViewerEngagementList.tsx`. The public viewer's floating chat toggle needed the _document owner's_ configuration, not the anonymous viewer's, so it's resolved server-side instead: `models/shareLink.ts#getByToken()` now returns a new `ai_chat_available` boolean (`u.ai_api_key_encrypted IS NOT NULL`, joined off the document's owner) alongside the rest of the public share response, and `ViewerPage.tsx` gates the toggle button on it.
- Phase 7 (Engagement, Trust & Growth) — Composite engagement score per viewer, surfaced as a new per-viewer list in the per-link analytics drawer, closing out Phase 7. Migration `018-add-downloaded-to-link-views.sql` adds `link_views.downloaded`, set (never unset) via `downloaded = downloaded OR $x` in `models/linkView.ts#upsertViewWithDedup` — tracked client-side by `ViewerPage.tsx`'s `hasDownloadedRef`, sent on the existing `beforeunload` beacon rather than firing an extra request per download. `models/linkView.ts#getViewersByLinkId()` groups `link_views` rows by fingerprint (a returning visitor outside the 30-min dedup window gets a new row, aggregated back together here) and `computeEngagementScore()` blends four signals into one 0-100 number: time on page and % of pages viewed (30% each, since finishing the document matters most), return visits and download (20% each) — 2 minutes of reading or 3+ visits each earn full marks on their own axis. A document with no known page count (non-PDF files) simply contributes 0 on the pages axis rather than guessing. `GET /api/v1/documents/[id]/links/[linkId]/analytics` now takes the document's `page_count` and returns a `viewers` array sorted highest-score-first; rendered by the new `components/analytics/ViewerEngagementList.tsx` in `LinkAnalyticsDrawer.tsx`, which also picked up the same scrollable-dialog treatment as the share-link modals below, since the added list pushed its content past the viewport on smaller screens.
